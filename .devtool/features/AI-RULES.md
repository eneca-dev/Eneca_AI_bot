# 🤖 Правила работы с Kanban для ИИ-ассистентов

Этот файл содержит инструкции для Cursor, Claude, GitHub Copilot и других нейронок по правильному взаимодействию с Markdown Kanban-доской в этом проекте.

## 1. Архитектура Kanban-доски
Канбан ведётся **не в одном файле**, а через создание индивидуальных `.md` файлов для каждой задачи (тикета) внутри папки `.devtool/features/`. Это позволяет избежать merge-конфликтов в Git при параллельной работе команды.

Каждый тикет должен содержать YAML-шапку (Frontmatter) и тело (описание задачи).

## 2. Команда и Инициалы
В команде 5 человек. При назначении исполнителя (поле `assignee`) используйте Имя Фамилию, а для генерации ID задачи — соответствующие инициалы:
- **Дмитрий Хуцишвили** -> `DH`
- **Вадим Тихомиров** -> `VT`
- **Владимир Нестерович** -> `VN`
- **Екатерина Зорина** -> `EZ`
- **Александра Бирило** -> `AB`

## 3. Названия файлов и заголовки (ID задач)
Категорически запрещено использовать абстрактные сквозные номера (вида `feature-809.md`), так как это вызывает конфликты при одновременном создании задач разными людьми.
**Формат ID и имени файла:** `[тип]-[инициалы]-[личный номер]`
- **Примеры:** `feature-DH-01.md`, `bug-VT-02.md`, `task-EZ-10.md`
- Личный номер сквозной **по человеку** (своя нумерация 01, 02, 03… у каждого исполнителя).
**Формат заголовка внутри файла:** В самом `.md` файле главный заголовок (H1) всегда должен начинаться с шифра задачи.
- **Пример:** `# bug-VT-02 Подготовка пайплайна тестирования`

## 4. Обязательная идентификация и привязка к задаче
- **Кто за рулём?** В начале каждого чата, если ИИ не знает, кто с ним сейчас работает, он **ОБЯЗАН спросить** имя пользователя и запомнить его. Это нужно, чтобы создавать и привязывать задачи под правильного человека.
- **Никаких выдумок.** Категорически запрещено придумывать или угадывать `assignee`. Если ИИ не уверен, кто сейчас работает — нужно спросить!
- **Работа только через задачи.** Любое действие с кодом или проектом должно происходить **только в рамках какой-либо зафиксированной задачи**.

## 5. Рабочий процесс (Workflow)
Мы **не используем** статус `in-progress` (он удалён из настроек).
Разрешённые статусы (поле `status`):
- `"backlog"` — задача в бэклоге.
- `"todo"` — задача в очереди на выполнение.
- `"review"` — код написан, ожидает проверки (Pull Request).
- `"done"` — задача завершена.

**Правила для статуса Review:**
При перемещении задачи в статус `"review"`, ИИ **обязан** дописать внутрь `.md` файла задачи два пункта:
1. Краткое описание того, как именно была решена исходная проблема.
2. Что конкретно требуется от ревьюера (на что обратить особое внимание при проверке).

## 6. Метаданные: Epic, Labels и Assignee
- **Epic:** Доступные значения для поля `epic` строго ограничены: `"bug"`, `"feature"`, `"test"`, `"refactor"`, `"chore"`.
- **Labels (Версии):** Поле `labels` **обязательно** должно содержать версию. ИИ должен вычислять версию из названия текущей Git-ветки (например, если ветка `v1.5.0`, то добавляем `"v1.5.0"` в `labels`). Если пользователь просит создать задачу "на следующую версию", ИИ должен сам увеличить мажорную/минорную версию (например, `"v1.6.0"`). Если текущая ветка не кодирует версию (напр. `dev`/`main`) — берём последний git-тег как текущую версию и при сомнении уточняем у пользователя.
- **Без исполнителя:** Если пользователь просит создать задачу без исполнителя, ИИ должен установить `assignee: null` и сообщить пользователю в чате: "задача создана, нет исполнителя".

## 7. Шаблон создания нового тикета
При просьбе пользователя создать новый тикет, ИИ должен **прочитать любой существующий `.md` файл задачи в этой папке** и скопировать из него структуру YAML-шапки (Frontmatter), подставив актуальные инициалы, статус, эпик, лейблы (версию) и текущее время. Не придумывайте новые поля из головы, используйте формат соседних файлов!

Набор полей frontmatter (только эти, новые не выдумывать):
`id, status, priority, assignee, epic, dueDate, created, modified, completedAt, labels, order`.

`created` / `modified` — реальное текущее время (UTC, ISO 8601 с `Z`), не выдумывать.

### Исключения (можно без тикета)
- Мелкие фиксы по запросу пользователя (опечатки, стили).
- Срочные hotfix по явной просьбе.
- Исследовательские задачи и вопросы.

## 8. Проектный аддендум: QA-сессии Teams-агента

Эти пункты специфичны для проекта `Eneca_AI_bot` и не противоречат разделам выше.

- **Scope.** Тестируем **только Teams-агента** (бэкенд протоколов встреч): онбординг
  бота, распознавание ссылки, подключение ассистента Recall, обработка записи
  (Whisper), генерация протокола, артефакты DOCX, доставка в Teams, персистентность
  в `meeting_reports`. Фронтенд `/meetings` (проект EnecaWork) — вне области.
- **Роли (Человек + Нейросеть).**
  - **Тестировщик (человек)** — выполняет ручные действия по сценарию.
  - **ИИ (организатор + проверяльщик)** — ведёт карточку, объясняет шаги, открывает
    нужные страницы в Playwright (где есть UI), отмечает чек-лист. После прохода
    сам читает логи / сеть / БД, формулирует ожидаемое поведение и сравнивает с
    фактическим.
- **QA-цикл на карточку:** объяснить шаг → тестировщик проходит → ИИ проверяет логи
  (нет лишних действий/повторов), сеть (нет дублей запросов), БД (`meeting_reports`
  сохранено корректно), фронт/Teams (отрисовано корректно) → сравнить ожидаемое vs
  фактическое → **баг** → отдельная карточка `bug-<инициалы>-NN` со ссылкой на тест;
  **бага нет** → тест в `done`.
- **Распределение:** один тест = одна область, не сваливать всё в одну мега-задачу.
