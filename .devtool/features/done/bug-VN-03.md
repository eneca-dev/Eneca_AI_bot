# bug-VN-03 Двойная обработка записи при дубль-доставке webhook (нет guard'а)

Найден в ходе расследования корня `23505` по [[test-VN-04]] (OBS-2). Исполнитель не назначен.

## Симптом
Если `recording.done` приходит дважды (две доставки), `_process_recording_with_whisper`
запускается дважды и **оба раза проходит весь пайплайн**: скачивание видео + Whisper + LLM.
`start_meeting_processing` у второго воркера ловит `23505` и **возвращает существующую строку,
но обработка НЕ прерывается** (server.py: после вызова сразу идёт `try: … Processing recording`).

Последствия:
- **двойная стоимость** обработки (2× Whisper + 2× LLM на одну запись);
- гонка двух `complete_meeting_report` (UPDATE) по одной строке;
- потенциально двойная доставка протокола в Teams (если у обоих воркеров есть `bot→conversation`).

## Как воспроизвели
В тесте в Recall было **два** активных webhook-эндпоинта (прод + ngrok) → каждый `recording.done`
уходил на оба сервера, оба писали в общую meetings-БД (см. OBS-2 в [[test-VN-04]]). Тот же эффект
даёт ретрай доставки самим Recall (Svix) на одном эндпоинте, если ответ запоздал/не 2xx.

## Важно про scope
Это **не** про конкретный тест-сетап с двумя эндпоинтами (операционно лечится удалением лишнего
эндпоинта). Это про устойчивость: при ЛЮБОЙ повторной доставке (включая штатные ретраи Recall)
система должна обрабатывать запись **один раз**.

## Предлагаемый фикс
Сделать `_process_recording_with_whisper` идемпотентным по состоянию строки: после
`start_meeting_processing` проверять возвращённую строку и **выходить**, если запись уже
обрабатывается/обработана другим воркером:
```python
row = meetings_db_client.start_meeting_processing(...)
# Если строка уже была (дубль доставки) и она уже done — выходим, не гоняем Whisper снова.
# Если processing — её ведёт другой воркер (или прерванный прошлый запуск). Решить по политике:
#   - done            -> skip (return)
#   - processing      -> skip, если строка свежая (например, по updated_at); иначе перезапуск
if row and row.get("status") == STATUS_DONE:
    logger.info(f"Recording {bot_id} already completed — skip duplicate processing")
    return
```
Тонкость: не блокировать легитимный **повтор после ошибки** (`status='error'` → переобработка
допустима). Возможна доработка `start_meeting_processing`, чтобы он явно сигналил «это новая
строка» vs «дубликат», чтобы воркер мог корректно решить.

## На что обратить внимание ревьюеру
- Не сломать повтор после `error` (переобработка должна оставаться возможной).
- Учесть, что `processing`-строка может остаться от упавшего прошлого запуска (нужен критерий
  «застряла» — напр. по `updated_at`/TTL), иначе зависшая строка навсегда заблокирует повтор.
- Добавить тест: дубль `recording.done` → ровно один проход Whisper/LLM.

## Операционная заметка (немедленно)
В дашборде Recall держать **один** webhook-эндпоинт на окружение. На время локальных тестов —
временно переключать на ngrok и **возвращать на прод** после, а не добавлять второй параллельный.
(Сделано: ngrok-эндпоинт убран, Recall и Azure возвращены на прод.)

## Решение (применено, → review)
1. **Как решено:** в `server.py` `_process_recording_with_whisper` теперь захватывает возврат
   `start_meeting_processing` и при `status == STATUS_DONE` **выходит до** скачивания/Whisper:
   ```python
   existing_row = meetings_db_client.start_meeting_processing(...)
   if existing_row and existing_row.get("status") == STATUS_DONE:
       logger.info(f"Recording {bot_id} already completed (status=done) — skipping duplicate processing")
       return
   ```
   Это покрывает реальный прод-сценарий: отложенный ретрай вебхука Recall **после** завершения →
   no-op (нет повторного Whisper+LLM, нет повторной доставки). `processing`/`error` НЕ скипаются —
   повтор после сбоя/застрявшего запуска остаётся возможен.
2. **На что смотреть ревьюеру:** регресс-тесты `tests/test_recall_webhook_idempotency.py`
   (`test_skips_processing_when_row_already_done`, `test_processes_when_row_is_processing`).
   Все suite зелёные (107 passed).

## Остаётся (по желанию, не блокер)
Полный guard на **одновременную** дубль-обработку (обе доставки видят `processing`) — требует
критерия «застрявшей» строки (TTL по `updated_at`). На практике этот случай возникал только при
двух эндпоинтах (артефакт теста, устранён операционно); штатные ретраи Recall — отложенные, их
закрывает skip-on-`done`. Можно оставить как отдельную доработку, если потребуется.
